サイトを更新したのに変わらない——CDNキャッシュでハマった話と再発防止

烏合ワークス ・ 2026-07-18

「公開処理は終わったのに、サイトを開くと前のまま」。自分たちの静的サイトで、実際にこの状況へ遭遇しました。

ファイルはクラウドストレージ上で新しくなっているのに、公開URLでは古いページが表示されます。原因は公開失敗ではなく、CDN（コンテンツ配信ネットワーク）に残ったキャッシュでした。

この記事では、原因をどう切り分けたか、更新フローへどんな再発防止を加えたかを紹介します。

ストレージは新しいのに、公開URLは古かった

サイトは、静的ファイルをクラウドストレージへ置き、その手前のCDNから配信する構成です。更新ファイルの同期は正常に終わり、ストレージ上の内容も新しくなっていました。それでもブラウザには古いページが出続けました。

シークレットウィンドウや別回線でも変わらなかったため、ブラウザだけのキャッシュではないと考え、HTTPレスポンスヘッダーを確認しました。

curl -I https://example.com/

CDNによってヘッダー名は異なりますが、CloudFrontなら Age や X-Cache が切り分けの手がかりになります。確認の結果、CDN側に保持された古い応答が配信されていると分かりました。

ファイル更新とキャッシュ更新は別工程

CDNは配信を速くするため、一度取得したファイルを一定時間保持します。そのため、配信元のファイルを更新しても、CDNのキャッシュがすぐ入れ替わるとは限りません。

自分たちの更新フローはストレージへの同期で終わっており、CDNへ古いキャッシュの破棄を依頼する工程がありませんでした。「ファイルを更新すれば表示も更新される」という思い込みが、手順漏れにつながりました。

更新フローへインバリデーションを追加した

CloudFrontには、指定したパスのキャッシュを無効化するインバリデーション機能があります。これを手作業ではなく、公開フローへ組み込みました。

  • ストレージへの同期が成功した後に、CloudFrontのキャッシュを無効化する
  • 自動化に与える権限を、対象のCloudFrontディストリビューションだけに絞る
  • 無効化が失敗した場合は、公開処理全体で把握できるようにする

導入後は、更新のたびにキャッシュ破棄が走るため、「公開は成功しているのに古いページが見える」という状態を避けられるようになりました。

費用は「リクエスト数」ではなく「パス数」で見る

AWS公式ドキュメントによると、CloudFrontのインバリデーションは、1つのAWSアカウント全体で毎月最初の1,000パスまで追加料金がかかりません。1,000パスを超えた分は課金対象です。また、ワイルドカードの /* は多数のファイルに一致しても1パスとして数えられます。

自分たちの更新頻度ではこの範囲に収まり、今回の変更による追加費用は発生しませんでした。ただし、無料なのは無制限ではありません。導入前にアカウント全体の利用量と更新頻度を確認する必要があります。

同じ問題に遭遇したときのチェックリスト

  1. 配信元のファイルが本当に更新されているか確認する。 更新されていなければアップロード工程の問題です。
  2. 別ブラウザや別回線で確認する。 ブラウザキャッシュだけの問題かを切り分けます。
  3. レスポンスヘッダーを見る。 CDNキャッシュの命中状況や経過時間を確認します。
  4. TTLとインバリデーション設定を確認する。 古いキャッシュが仕様どおり残っている可能性があります。
  5. 公開フローへキャッシュ更新を組み込む。 手作業に依存すると、次の更新でまた忘れます。
  6. 権限と費用上限を確認する。 自動化には必要最小限の権限だけを与え、無料枠を超える可能性も把握します。

インバリデーションだけが正解ではない

更新頻度が高い静的ファイルでは、ファイル名へバージョンを付ける方法や、適切なCache-Controlを設定する方法もあります。AWSも、頻繁に更新することが事前に分かっているファイルでは、バージョニングや短い有効期限を検討するよう案内しています。

今回のサイトでは規模と更新頻度を踏まえ、公開時のインバリデーションを採用しました。運用規模が大きくなれば、HTMLと画像・CSS・JavaScriptでキャッシュ戦略を分ける方が効率的です。

まとめ

CDNを使う静的サイトでは、「配信元の更新」と「閲覧者に届く表示の更新」は別工程です。更新したのに変わらないとき、すぐに公開失敗と決めつけず、ストレージ、ブラウザ、CDNの順に切り分けると原因へ早くたどり着けます。

そして原因がキャッシュなら、毎回の手作業ではなく、権限と費用を確認したうえで公開フローへ組み込むのが再発防止になります。

参考：AWS公式ドキュメント「Invalidate files」「Pay for file invalidation」（2026年7月18日確認）。料金・無料枠は変更される可能性があるため、導入時に最新情報をご確認ください。
